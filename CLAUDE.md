# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```
pnpm install                    # install deps (use over adding deps by hand)
pnpm dev                        # web UI dev server (http://localhost:5173)
pnpm run typecheck              # tsc --noEmit in every package
pnpm run test                   # vitest run in every package (single run)
pnpm --filter @rebalancer/solver test -- rebalance   # run a single test file/pattern
pnpm run test:watch             # vitest in watch mode, solver package only
pnpm run build                  # compile every package to dist/
pnpm icons                      # regenerate raster icons in apps/web/public from their SVGs (sharp)
pnpm solve -p <scenario.json> [--sell] [--sell-taxable] [-c accountId:amountCents ...]   # run the CLI
```

Node version is pinned in `.nvmrc` (24); run `fnm use` before working. CI (`.github/workflows/ci.yml`) runs `pnpm install --frozen-lockfile && pnpm run typecheck && pnpm run test` on every push.

## Architecture

pnpm workspace (`pnpm-workspace.yaml`) with three packages:

- **`packages/solver`** — the rebalancing engine. Pure TypeScript: no DOM, no `fetch`, no filesystem access, no network calls. Deterministic — same input always produces the same output. Its public exports are `rebalance()`, `validateScenario()`, `DEFAULT_TOLERANCE_BPS`, and the domain types in `src/types.ts`; nothing else in `src/` should be imported from outside the package.
  - `src/rebalance.ts` — orchestration: validates input, reduces it to a transportation problem, delegates placement to `allocateLp()`, translates the resulting per-(account × fund) deltas into buy/sell `Trade`s with human reasons, and formats warnings. The behavior contract (buy-only by default, opt-in selling with tax guards, governed by a tolerance band) is explained in a comment block at the top. It also does all semantic input validation (targets and per-fund class weights summing to 10000 bps, referential integrity of non-empty ids, unique fund tickers and asset-class/account names, duplicate-free fund menus, non-negative integers, and a ~$9B portfolio-total cap that keeps cents × bps math inside Number.MAX_SAFE_INTEGER) — the solver is the trust boundary, so callers don't need to re-validate.
  - `src/allocate.ts` — the allocator seam: the `TransportationProblem`/`Allocation` contract, purely in (account × fund) space — ids and integer cents; no names. Demands stay per asset class; each fund's `weights` map (assetClassId → bps) connects fund positions to class exposures, so a blended fund's buys/sells move all its components in lockstep. Money never leaves an account, so each account's post-trade total is fixed (a fixed-supply transportation problem). (A greedy waterfall engine used to live behind this seam, selectable via a removed `options.optimizer`; it's in git history if ever needed.) Do not hand-roll a min-cost-flow solver.
  - `src/allocate.lp.ts` — the engine, using YALPS (pure JS, synchronous — the async/WASM LP libraries would break `rebalance()`'s sync purity). Variables are per-(account × fund) final values, so blended funds are handled natively (it can even sell VT and buy back the intl slice with VXUS to shed only a US excess), and it can *relocate* a class between accounts when restricted fund menus require it (see the "restricted fund menus" tests — a hard requirement). Lexicographic objectives (deviation → total sells → taxable sells → tax-preferred placement → fund-preference order; `options.optimizeAssetLocation` promotes tax-preferred placement above the two sell stages, so the solver trades *in order to* fix asset location — still never at the allocation's expense, and still behind the selling guards), each pinned as a constraint before the next solve; the last stage makes buys pick the earliest `availableFundIds` entry and sells drain the latest, and can't add churn because total sells are already pinned. `minTradeCents` is honored by iterative refinement: ban selling any position whose sell lands below the floor, re-solve, repeat. Simplex floats are rounded back to integer cents by snapping (near-)untouched positions to their current value (so noise never fabricates a one-cent trade) plus a per-account largest-remainder repair: account totals stay exact; class totals can carry < 1 cent of noise per position, which the property tests carry as explicit slack. The tolerance band is an *eligibility* test computed from the inputs in exact integer cent-basis-points: beyond-band classes are pulled to their exact target, within-band classes are frozen against selling — for a blend, the class-level floor is what blocks (or permits) selling it. `src/allocate.property.test.ts` brute-forces every small problem it can enumerate (constrained single-account, unconstrained multi-account, and constrained two-account) and holds the LP to the true optimum, modulo the documented rounding slack.
  - `src/scenario.ts` — `validateScenario(unknown): Scenario` structurally validates an untrusted parsed JSON document (field presence, primitive types, enum membership, unknown-key rejection; keys starting with `_` are ignored as comments). Semantic rules stay in `rebalance()`.
  - `fixtures/example.json` (buy-only golden: a hand-invented household — taxable brokerage, two IRAs, a 401(k), an HSA) and `fixtures/sell-required.json` (selling golden: a drifted portfolio only fixable by selling) are placeholder data, not real portfolios. Both are complete `Scenario` documents — the same shape the CLI reads.
  - `package.json` `main`/`types`/`exports` point directly at `src/index.ts` (not `dist/`) so workspace consumers (the CLI, via `tsx`) don't need a build step in dev. `pnpm run build` still produces a real `dist/` for whenever this is published.
- **`apps/web`** — React (Vite) UI over the solver; see `WEB_UI.md`. One `Scenario` object in a single `useState` is the entire state: no backend, no URL state, and **no localStorage/sessionStorage** — persistence is the user downloading/uploading the canonical Scenario JSON (same document the CLI reads; downloads carry an ignored `"_format"` comment key). Hard constraints, same spirit as the solver's: the UI imports only `@rebalancer/solver`'s public API and computes nothing about money — no rollups, gap math, or placement decisions in React code (if the UI needs a derived number, the solver should export it); money stays integer cents end-to-end (typed text is parsed textually in `parse.ts`, never via float multiplication; dollars appear only at render via `format.ts`); no `fetch`/network (market values come from user input, always); no HTML `<form>` submits. Structure: `PortfolioEditor`/`ScenarioEditor` build the scenario through pure updaters in `scenario-edit.ts` (removals cascade to keep referential integrity — the solver revalidates anyway and its error message is shown in place of results), `App` runs `rebalance()` on every edit, `ResultView` is pure presentation mirroring the CLI's output structure. Tests: vitest + testing-library under jsdom (`globals: true` is required for testing-library's auto-cleanup).
- **`apps/cli`** — thin wrapper around the solver, run with `tsx` and `node:util`'s `parseArgs`. It only imports `@rebalancer/solver`'s public API (never reaches into `packages/solver/src` directly), and does all the I/O the solver isn't allowed to do (reading the scenario JSON file from disk). `-p` loads a whole `Scenario` (portfolio + targets + contributions + options); flags are overrides (`-c` replaces the file's contributions; `--sell`, `--sell-taxable`, `--optimize-location`, `--tolerance-bps`, `--min-trade-cents` adjust options).

Money is always an integer number of cents (never a float). Weights/targets are always integer basis points 0–10000 (never a 0–1 fraction).

- Tests are colocated as `*.test.ts` next to the code they cover, run with vitest.
- Property tests use `fast-check`: `src/rebalance.property.test.ts` (no sells without `allowSelling`, per-account conservation `buys − sells == contribution`, never selling a class below target, taxable-sell guard, blended-fund invariants, determinism under shuffled input ordering) and `src/allocate.property.test.ts` (the LP matches a brute-force minimum-deviation reference on small constrained problems, and reaches target exactly when unconstrained).
- Each package's `vitest.config.ts` restricts test discovery to `src/**/*.test.ts` — without this, vitest also picks up compiled `*.test.js` files under the git-ignored `dist/`, double-running every test after a local build. Keep this scoping if touching vitest config.
- TypeScript (`tsconfig.base.json` at the repo root, extended per-package) uses `module: nodenext` + `verbatimModuleSyntax`, so relative imports must include the `.ts` extension (e.g. `import { rebalance } from "./rebalance.ts"`) — this is intentional, not an error to "fix". `allowImportingTsExtensions` + `rewriteRelativeImportExtensions` are on so this works both for `tsc --noEmit` and for real builds (which rewrite `.ts` → `.js` in emitted output).

## Domain model notes (not obvious from the code alone)

- `Contribution { accountId, amount }` is cash **earmarked to a specific account** — money never moves between accounts (mirrors reality: a 401(k) payroll contribution can't land in an IRA). The CLI's `-c` flag is therefore `accountId:amountCents`, repeatable, not a single bare number. The same constraint applies to selling: cash raised by a sell can only buy funds in that same account, which is what keeps the allocation problem account-local.
- `Fund.assetClasses` is a map assetClassId → integer basis points that must sum to exactly 10000. Most funds are `{ one_class: 10000 }`; a blend like VT is `{ us_stocks: 6500, intl_stocks: 3500 }`. A blend's value counts toward each component class proportionally (reports split integer cents by largest remainder), and trading it moves every component in lockstep — there is no way to buy just the US slice of VT, which is exactly what the per-fund LP encodes. Zero-weight entries are allowed (the web UI keeps them while a blend is mid-edit) and treated as absent.
- `Account.availableFundIds` is **ordered, most-preferred first**: buys pick the earliest fund of the target asset class, and sells liquidate the *least*-preferred fund of the class first (a held fund absent from the list — "held but no longer buyable" — is treated as least preferred of all). There is no separate fund-preference field. In the LP this is the final lexicographic stage (minimize Σ rank·dollars), so it only breaks ties the earlier objectives left open. Notably, **surplus contribution cash** (cash with no reachable gap) is placed by asset-class `taxPreference` first — tax-preferred placement is stage 4, fund order stage 5 — so `availableFundIds[0]` receives surplus only when no tax preference applies. That priority (and "leave surplus as cash" as an alternative) is a policy the user currently can't choose; see the note atop allocate.lp.ts's stage list before changing it.
- `AssetClass.taxPreference` (`prefer_taxable` | `prefer_tax_advantaged` | `neutral`) ranks which accounts should be preferred when more than one is eligible to receive a given asset class (e.g. bonds default to `prefer_tax_advantaged`). This field isn't in most "textbook" domain models — it exists because the household in `fixtures/example.json` needed a simple way to express asset-location preference without a separate ranking table. By default it is only a tie-break among trades already needed — an on-target portfolio is never traded no matter how badly located. `options.optimizeAssetLocation` (CLI `--optimize-location`) promotes it to an objective: the solver then relocates classes toward their preferred account types via mirrored sell/buy pairs (money still never leaves an account), which needs `allowSelling`, plus `sellInTaxableAccounts` to move a class *out of* a taxable account. When location mode is on but only the taxable-sell guard stands between the user and better placement, `rebalance()` warns with the exact dollars blocked per class — established by a counterfactual re-solve with the guard lifted (never a heuristic), so it can't fire when relocation is impossible for other reasons. Sells have their own rule: prefer tax-advantaged accounts (no tax consequence), and never sell in taxable accounts at all unless `sellInTaxableAccounts` is set.
- The tolerance band (`toleranceBps`, default 50) is the stand-in for the NP-hard "minimize number of trades" objective: a class within ±band of target is treated as on-target — not bought toward, not used as a sell donor, not "fixed" by selling, not warned about. `minTradeCents` floors sell-funded moves only; contribution cash is always fully invested because cash may not sit idle in an account.
